## Required Screens

### 1. Home Screen
- active role
- current task
- next expected response type
- visible protocol status

### 2. Relay Screen
- selected role: Vale / Veris / Leora
- input area
- output area
- generated label block
- copy-ready text

### 3. Issue Intake Screen
Required fields:
1. What you did
2. What happened
3. What should have happened
4. Scope

No submission path if any field is empty.

### 4. Task Board Screen
Lanes:
- Now
- Next
- Blocked
- Delegated

### 5. Memory Screen
Shows connected files/locations:
- `PROJECTS_IN_PROGRESS.md`
- `CONTEXT_LOG.md`
- `PRESERVATION\CODEX`

For MVP, this screen should show read/write targets and operation status clearly.

## Required Data Files

### protocol_rules.json
Must support at least:
- relay_message
- issue_intake
- ritual_of_alteration
- audit_report

Each rule should define:
- required fields
- required order
- blocking message
- output template

### role_definitions.json
Must define:
- Vale
- Veris
- Leora

Each role must include:
- name
- role summary
- expected deliverables
- valid task types
- invalid task types

## Label Format
This must be fixed before implementation starts.

Default MVP format:
- `TO: <ROLE>`
- `FROM: <ROLE or TIMOTHY>`
- `TYPE: <ACTION TYPE>`
- `SUBJECT: <SHORT SUBJECT>`

This format should be generated by the app, not improvised manually.

## Blocking Rules
MVP must block:
- unlabeled relay
- mixed-agent paste without source marker
- repair request without issue intake
- build request routed to the wrong role
- approval/send attempt without explicit next step
- report submission missing required protocol sections

## Context Log Write Discipline
If the app writes to `CONTEXT_LOG.md`, it must append using the existing date-header plus section-block pattern already in use.

The app must not:
- rewrite old entries
- invent a new log structure
- silently alter existing records

For MVP:
- prefer append-only helper functions
- show preview before write
- require explicit confirmation before write

## Memory Hook Rules

### PROJECTS_IN_PROGRESS.md
- Append/update only through explicit user-confirmed action
- No autonomous restructuring

### CONTEXT_LOG.md
- Append only
- No overwrite
- Use existing format

### PRESERVATION\CODEX
- Read allowed
- Write only through explicit preservation actions
- No silent edits

## Build Sequence
1. Scaffold Tauri + React + Vite app.
2. Add Tailwind.
3. Implement static role definitions.
4. Implement `protocol_rules.json`.
5. Build Session Router UI.
6. Build Protocol Gate engine.
7. Build Issue Intake form.
8. Build Task Board.
9. Build memory hook stubs.
10. Test real relay traffic.

## Acceptance Criteria For Alpha
- Timothy can route one Vale message and one Veris message without losing who said what.
- App blocks one incomplete issue-intake attempt.
- App generates a correct labeled relay block.
- App visibly distinguishes active role and expected response type.
- No writes occur silently.

## Explicit Non-Goals
- No browser automation
- No autonomous sending
- No multi-agent orchestration
- No cloud sync
- No analytics
- No Unreal integration in MVP
